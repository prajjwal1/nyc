## Proposals (ordered by value)

### U1: Add relative-day scent to cards in hero sections  [BEHAVIOR-CHANGING, low risk]
- **Metric moved**: high-conviction event ratio (perceived) + required-detail surfacing.
- **Component(s)**: `site/app/components/EventCard.tsx` FeedCard, around the time row at lines 175-196.
- **localStorage key**: none.
- **Problem**: The Tonight / Following / Saved / Just-Added heroes render cards with NO date. A followed event tomorrow and one 3 weeks out are visually identical. The single most-wanted glance detail ("when, relatively") is missing exactly in the sections that matter most for the North Star.
- **Change sketch** — add a `dayLabel` computed in FeedCard and render it as the first pill in the meta row (before the clock):
  ```tsx
  // near top of FeedCard, after openedFeed
  const dayLabel = relDay(event.date); // "Tonight" | "Tomorrow" | "Sat" | "Jul 12" | null
  // ...in the meta row, before the timeStr span:
  {dayLabel && (
    <span className="font-semibold text-gray-700">{dayLabel}</span>
  )}
  ```
  ```tsx
  // module-scope helper
  function relDay(iso: string): string | null {
    try {
      const today = new Date(); today.setHours(0,0,0,0);
      const d = new Date(iso + "T00:00:00");
      const diff = Math.round((d.getTime() - today.getTime()) / 86400000);
      if (diff < 0) return null;
      if (diff === 0) return "Today";
      if (diff === 1) return "Tomorrow";
      if (diff <= 6) return d.toLocaleDateString("en-US", { weekday: "short" });
      return d.toLocaleDateString("en-US", { month: "short", day: "numeric" });
    } catch { return null; }
  }
  ```
- **Rationale**: Makes the highest-conviction sections (Following/Saved/Tonight) answer "when can I actually go?" at a glance — the core North-Star question.
- **Risk**: In the dated date-group list of TopPicks (lines 437-475) and in EventList, the relative day duplicates the group header. Acceptable (a small "Today"/"Tomorrow" pill next to time is not noise), but if the orchestrator wants zero duplication, gate it with a `showDay` prop that TopPicks/EventList pass `false` for grouped lists and `true` for heroes. Recommend the prop approach — ~6 extra lines, cleaner.

### U2: De-duplicate + clean the location line  [BEHAVIOR-CHANGING, low risk]
- **Metric moved**: required-detail surfacing / clutter reduction.
- **Component(s)**: `site/app/components/EventCard.tsx` lines 182-195.
- **Problem**: `name` frequently already contains the neighborhood ("Book Club Bar East Village…"), and the card appends "· east village" again. 8 rows have a name/neighborhood conflict that shows contradictory text ("Bushwick, 380 Troutman Street · east village").
- **Change sketch** — suppress the neighborhood suffix when the name already contains it (case-insensitive substring), and trim an embedded street-address tail from the displayed name so the badge is the clean signal:
  ```tsx
  const nb = event.location.neighborhood?.trim();
  const nameLc = (event.location.name || "").toLowerCase();
  const showNb = nb && !nameLc.includes(nb.toLowerCase());
  // ...
  {event.location.neighborhood && showNb && (
    <span className="text-gray-400 shrink-0">· {nb}</span>
  )}
  ```
- **Rationale**: neighborhood is a top attend-decision detail; showing it once, cleanly, raises scent without adding chrome.
- **Risk**: Purely display; conservative substring guard. Does not touch the underlying data bug (that's a normalizer concern, flag to the pipeline agent separately).
